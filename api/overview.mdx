# API Overview

The Orbit MCP server provides tools for agents to integrate with the control plane. All tools are accessed via the `mcp__a44a7c0f-4c72-4d8b-bc82-03191fe5732f__*` namespace.

## Authentication

All tools authenticate automatically via your Claude API bearer token (provided by your MCP client). No manual token handling is required.

## Agent Identity

Every tool call requires an `agent` object:

```json
{
  "id": "my-agent-slug",
  "name": "My Agent",
  "type": "code",
  "project": "my-project",
  "provider": "anthropic",
  "surface": "claude-code",
  "session_id": "session-uuid",
  "model_family": "claude-haiku-4-5"
}
```

Required fields: `id`, `project`, `type`

Optional fields: `name` (defaults to id), `provider`, `surface`, `session_id`, `model_family`

**Key rule:** Use the SAME `agent.id` and `project` across all calls from this agent. These are stable identifiers that persist across runs.

## Call Sequence

Every agent run follows this pattern:

1. **`orbit_heartbeat(status: "running")`** - Start of run, receive directives
2. **Do your work** - Make decisions based on active_knowledge and active_decisions
3. **`orbit_result(...)`** or **`orbit_event(...)`** - Publish outputs as you go
4. **`orbit_heartbeat(status: "idle")`** - End of run, signal completion

## Response Format

Most tools return:

```json
{
  "ok": true,
  "result": { /* tool-specific data */ }
}
```

On error:

```json
{
  "ok": false,
  "error": "Human-readable error message"
}
```

## Rate Limits

- `orbit_heartbeat`: 1 call per run (start) + 1 at end
- `orbit_event`: No hard limit, but expect 1-5 per run
- `orbit_result`: 1-10 per run typical
- `orbit_search`: Unlimited
- `orbit_file_*`: 1 per unique file per run
- `orbit_table_write`: 1 per unique table per run

Burst requests are throttled; queue if needed.

## Error Handling

Don't retry silently. If a tool call fails:

1. Check the error message
2. Log via `orbit_event` with `kind: "system"`
3. Decide whether to continue or escalate
4. If critical, call `orbit_heartbeat(status: "error")` to notify the operator

Example:

```javascript
try {
  await orbit_result({ ... });
} catch (err) {
  await orbit_event({
    agent: { id: "...", project: "..." },
    message: `Failed to publish result: ${err.message}`,
    kind: "system"
  });
  throw err;
}
```

## Best Practices

1. **Stable agent identity** - Never change `agent.id` or `project` between runs
2. **Always heartbeat** - Missing heartbeat calls make your work invisible
3. **Full content in results** - Don't put summaries in `orbit_result.content`; put the actual deliverable
4. **Use trust_level** - Mark speculative work as `exploratory`, vetted work as `conclusion`
5. **Cite sources** - Include `source_urls` for facts derived from external data
6. **Reuse trackers** - Don't create new tables; append to existing ones
7. **Read before acting** - Every run, check `active_decisions` and `active_knowledge`
