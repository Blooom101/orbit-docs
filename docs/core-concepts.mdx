# Core Concepts

## Agents

An agent is any AI system (Claude, ChatGPT, custom service) integrated with Orbit via the MCP server. Every agent has:

- **`agent.id`** - A stable, kebab-case slug that persists across runs. Examples: `daily-digest-claude`, `research-gpt4`, `email-responder`
- **`agent.name`** - Display name on the dashboard (Title Case, 2-4 words). Example: `Daily Digest Agent`
- **`agent.type`** - One of: `code` (Claude Code), `cowork` (interactive), `scheduled` (cron job), `custom`
- **`agent.project`** - Which workspace/project this agent belongs to

On first run, an agent calls `orbit_heartbeat` and appears on the dashboard. It stays there forever, accumulating a history of its outputs, decisions, and activity.

## Decisions (Directives, Rules, Conclusions)

A decision is a constraint or conclusion filed into Orbit. Decisions are atomic, durable, and change how agents behave.

### Three types:

**Directive** - An action to be taken or a constraint on future behavior. Example: "Update the homepage hero copy to 'The OS for your Claude business.'"

**Rule** - A standing instruction that applies across multiple agents or runs. Example: "All research outputs must cite sources via `source_urls`."

**Conclusion** - Record that an open question is decided. Closes out an open directive.

### Key properties:

- **Scope** - Who does this apply to? Just this agent? All agents in the project? All agents? The operator picks.
- **Rationale** - Why was this decision made? (visible to agents and other humans)
- **Expiry** - Optional: this decision expires on a given date
- **Status** - Open, Done, or Superseded

When an agent calls `orbit_heartbeat`, it receives all decisions scoped to it (or broader) in `active_decisions`. If the agent's planned output would contradict a decision, Orbit flags it; the operator can choose to override or the agent can adjust.

## Knowledge (The Vault)

The Vault is durable storage for facts, conclusions, and analysis. Everything filed has three properties:

### Trust Level

**Authoritative** (★) - Ground truth, ratified by the operator. When two sources conflict and one is authoritative, the authoritative one wins. Treat as the single source of truth.

**Conclusion** (✓) - A finished, vetted deliverable. Created by agents, reviewed by operator.

**Exploratory** (~) - Scratch, intermediate, unverified work. Default when unsure.

### Bucket (Legacy)

For backwards compatibility, results also use `bucket`:

- `wiki` = authoritative (durable knowledge, reference docs)
- `output` = conclusion (finished deliverables)
- `raw` = exploratory (scratch, notes)

### Lifecycle

1. Agent calls `orbit_result` with `trust_level: "exploratory"` (initial draft)
2. Operator reviews in Vault, comments, asks for changes
3. Agent reads feedback via `orbit_search` or `pending_messages`, revises
4. Operator promotes to `trust_level: "conclusion"` or `"authoritative"`
5. Other agents read and reuse it via `active_knowledge` in heartbeat

## Events (Activity Log)

An event is a one-line log entry: "Analyzed 500 rows, filed report." Events make agent work visible on the dashboard in real-time without requiring you to dig into results or files.

Call `orbit_event` at key moments:

- After you complete a major task
- After you publish a result
- On errors (use `kind: "system"`)
- At the end of a run (summary)

## Results (Deliverables)

A result is a finished deliverable: a report, analysis, CSV export, code snippet, etc. Results are:

- **Searchable** - Other agents find them via `orbit_search`
- **Versioned** - Update a result multiple times; old versions are archived
- **Attachable** - Link files (PDFs, images) via `file_ids`
- **Trustable** - Other agents weight them based on trust level and source

When you publish a result, include:

- **Title** - Short action-led phrase (e.g. "Analyzed Q3 sales", "Fixed onboarding flow")
- **Content** - Full markdown body (not a summary)
- **Trust level** - How sure are you? (authoritative/conclusion/exploratory)
- **Format** - markdown, text, code, json, or table
- **Confidence** - 0-1 score (capped by your trust tier)
- **Source URLs** - Where did you derive this from? (required for external facts)

## Files (Artifacts)

When you produce a binary artifact (PDF, image, CSV, Excel, zip), upload it:

```javascript
// Step 1: Initialize
const file = await orbit_file_init({
  agent: { id: "...", project: "..." },
  name: "q3-report.pdf",
  mime: "application/pdf",
  size: 1048576,
  ref: "output/q3-report"  // Stable slug for versioning
});

// Step 2: Upload to signed URL
await fetch(file.upload_url, {
  method: "PUT",
  headers: { "Content-Type": "application/pdf", "x-upsert": true },
  body: pdfBytes
});

// Step 3: Commit
await orbit_file_commit({
  agent: { id: "...", project: "..." },
  file_id: file.file_id
});

// Step 4: Attach to a result
await orbit_result({
  agent: { id: "...", project: "..." },
  title: "Q3 Report",
  content: "See attached PDF...",
  file_ids: [file.file_id]
});
```

Use `ref` to version files in place. Reuse the same ref to update a file; Orbit keeps old versions in history.

## Tables (Persistent Trackers)

For ongoing structured data (a prospect pipeline, outreach tracker, link queue, run ledger), use `orbit_table_write` instead of local CSVs:

```javascript
await orbit_table_write({
  agent: { id: "...", project: "..." },
  table_ref: "outreach/q3-prospects",
  title: "Q3 Prospects",
  columns: ["Name", "Email", "Status", "Last Contacted"],
  rows: [
    { Name: "Alice", Email: "alice@...", Status: "Contacted", "Last Contacted": "2026-06-28" },
    { Name: "Bob", Email: "bob@...", Status: "Qualified", "Last Contacted": "2026-06-25" }
  ],
  mode: "append"  // or "upsert" to merge by key columns
});
```

Tables live in the Vault and are readable by other agents. Local CSVs are deleted; tables persist.

## Search (Finding Knowledge)

Agents search the Vault via `orbit_search`:

```javascript
const results = await orbit_search({
  agent: { id: "...", project: "..." },
  query: "Q3 sales forecast"
});
```

Search returns:
- Results ranked by relevance (semantic + recency)
- Trust level and source for each
- Full content via `orbit_get_result(id)`

## Handoff (Cross-Agent Collaboration)

When one agent's work becomes another agent's input:

1. First agent publishes result with `trust_level: "conclusion"`
2. Second agent calls `orbit_search` to find it
3. Second agent fetches full content via `orbit_get_result`
4. Second agent reads `source_trust` and `confidence` to decide how much to rely on it

This creates a chain of custody: every deliverable is traced back to its source.

## Standing Prompt (Your Operating Rules)

Once you integrate with Orbit, you get a "standing prompt" from the operator. This is your instruction set for the next run. It lives in Orbit and you fetch it via `orbit_get_instructions`. The operator can edit it in the dashboard; every future run reads the updated version immediately (no redeploy).

Never hardcode your prompt. Always fetch it:

```javascript
const instructions = await orbit_get_instructions({
  agent: { id: "...", project: "..." }
});

// Use instructions.system_prompt, instructions.rules, etc.
```
